## Post-Phase-1 — Meet Front Office color, and draft/Apply save model

User feedback after Phase 1: (1) "Meet Front Office" should be red, not orange; (2) every action (drag, edit, delete) currently saves to the server immediately — wanted instead: edits accumulate in a temporary state you can step away from and back into, with an explicit "Apply"/Save action that makes them permanent, and a confirmation before overwriting a previously-saved day's schedule on a later re-save.

- **Color**: `time-block--orange` in `ScheduleGrid.css` changed from `#ed8f3c` to `#d64545` (red). The internal status value is still `"orange"` (matches the DB enum, the project plan's terminology, and existing data) — only the swatch changed.
- **Draft/Apply save model** — confirmed two design questions with the user first (where the draft lives; whether confirmation fires per-edit or once on save) before touching anything, since this is a real architecture change:
  - **Schedule model**: added `draftRows` alongside the existing `rows`. `rows` is now strictly the *published* schedule; `draftRows` is the working copy every edit applies to. Nothing reaches `rows` until an explicit apply.
  - **Migration safety**: `getSchedule` backfills `draftRows = rows` the first time it reads a document created before this field existed, then persists that backfill. Verified directly against the user's own real, currently-in-progress data (a row "infant - jenifer" with a green + an orange block on `2026-06-23`, found live in the database while building this) — confirmed `rows` was untouched and `draftRows` was correctly populated to match.
  - **Backend**: `addRow`, `addOrUpdateBlock` (via the conflict-check middleware), `deleteBlock`, `deleteRow` now all read/write `draftRows` instead of `rows`. New `POST /api/schedule/:date/apply` (`applyDraft` controller) copies `draftRows` → `rows`, preserving existing row/block ids (not regenerating them) so future edits-by-id keep working after a publish. `copySchedule` now copies the previous day's *published* `rows` into the new date's `draftRows` (pending its own apply) rather than writing `rows` directly, and now requires the previous day to actually have published content (`rows.length > 0`), not just any document.
  - **Frontend**: `useSchedule` exposes `isPublished` (has `rows` ever been applied for this date), `isDirty` (does `draftRows` differ from `rows`), and `applyAndSave()`. `ScheduleGrid` now renders and edits `schedule.draftRows`, never `schedule.rows`, directly. `SchedulePage` gained a status indicator ("All changes saved" / "Unsaved changes") and a "Save Schedule" button, disabled when not dirty. Clicking it: if nothing has ever been published, saves directly (nothing to overwrite); if something was already published and the draft has since changed, shows the new `SaveConfirmModal` ("you've edited this — overwrite the saved version?") before applying. Because the draft now lives server-side, navigating away to the calendar and back no longer loses anything — that need is satisfied by the storage choice itself, no extra client-side persistence required.
  - **Calendar dot indicator** and the **copy-previous-day eligibility check** both switched from "does a schedule document exist" to "does it have a published (`rows.length > 0`) schedule" — an in-progress, never-applied draft shouldn't look like a saved day or be offered as something to copy from.
  - Verified end-to-end against the live backend on a throwaway date: new row+block lands in `draftRows` only, first apply publishes it, further edits make the draft diverge from published (dirty), re-apply overwrites published with the new draft, conflict detection still correctly scopes to the draft. Confirmed via curl that the running dev server is serving all the new frontend code. Lint/build clean.
  - Not yet verified in the browser — same standing Chromium/sudo limitation as always; this is a significant enough change that the user's own click-through matters more than usual here.

## Real browser testing became possible mid-session — two real bugs found and fixed

The standing "no browser available" blocker (see Blockers, the `libnspr4.so` issue) turned out to be workable without `sudo`: Playwright's Chromium binary was already downloaded from earlier in the project (`~/.cache/ms-playwright`), and its missing shared libraries (`libnspr4`, `libnss3`, `libasound2t64`) could be fetched with `apt-get download` (download-only doesn't need root) and extracted locally with `dpkg-deb -x` (no install, no root) into `/tmp/localdeps`. Pointing Chromium at that directory via `LD_LIBRARY_PATH` makes it launch normally. A small Playwright driver script (`executablePath` pointing at `~/.cache/ms-playwright/chromium-1228/chrome-linux64/chrome`, same `LD_LIBRARY_PATH`) gives full real-browser access: navigation, clicking, dragging (via `page.mouse`), screenshots, and console/error capture. **This should be reused in future sessions instead of assuming browser testing is impossible.**

Used this to actually click through the app for the first time (calendar, the real `2026-06-23` data read-only, then full interactive flows on throwaway dates `2026-12-15` through `2026-12-18`, all cleaned up afterward). Found and fixed two real bugs that no amount of API-level scripting had caught, because they were specifically about the *frontend* reading the *response shape* of the draft/Apply rework:

1. **`AssignmentDropdown.handleConfirmTeacherClassroom` read `updatedAfterRow.rows.find(...)` after calling `addRow`, instead of `.draftRows`.** Since `addRow` puts the new row into `draftRows` (per the rework), `.rows` was always empty for a brand-new pairing, `.find()` returned `undefined`, and `newRow.rowId` threw — caught and shown as a generic "Something went wrong" toast. This broke creating the very first assignment on any date through the UI (the placeholder-row bootstrap flow) — about as core a path as exists in this app. Fixed: read `.draftRows` instead. Confirmed working afterward by repeating the exact same browser flow on a fresh date.
2. **`useSchedule`'s `isDirty` compared `JSON.stringify(draftRows) !== JSON.stringify(rows)`.** The server serializes `rows` (manually rebuilt by `/apply` with `rowId` first) and `draftRows` (a Mongoose subdocument, schema key order) with different key orders even when the content is identical — so right after a successful save, the two stringified forms still differed and `isDirty` stayed permanently `true`. The "Unsaved changes" / "All changes saved" indicator and the Save button's disabled state never reflected reality once anything had ever been saved. (This exact key-ordering quirk had already been hit once before in this session during E2E test assertions and dismissed as a test artifact — turns out it had a real production consequence on the client.) Fixed with a `canonicalizeRows` helper that rebuilds both sides with an explicit, fixed key order before comparing.

Both fixes verified in the real browser, not just via API script: full flow on a fresh date — create row+block (Unsaved changes) → first save (All changes saved) → delete the block (Unsaved changes again) → Save → **re-save confirmation modal correctly appears** → confirm → All changes saved. Zero console errors throughout (aside from the expected 404s from the existence-check pattern). Also confirmed: the calendar page alone generates ~30 console "Failed to load resource" 404s per month view (one per day's existence probe) — functionally harmless (caught, handled) but a real console-cleanliness gap given the "zero console errors" standard; not fixed, flagging as a known, accepted cost of the no-list-by-month-endpoint design noted back in Layer 7.

Visually confirmed via screenshots: calendar today-highlight + dot indicator, the real `2026-06-23` data rendering correctly (green active shift + **red** Meet Front Office block, confirming the earlier color fix), 12-hour time labels with AM/PM only at the boundaries, the "6 PM" end-cap, disabled Save button when not dirty, the Teacher/Classroom panels with working inline add and auto-select, the exact confirmation-summary wording from the spec, and the block context menu.
